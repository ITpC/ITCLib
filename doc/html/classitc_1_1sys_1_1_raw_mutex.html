IEEE Std 1003.1-2001 specification says: Implementations are required to allow an object to be destroyed and freed and potentially unmapped immediately after the object is unlocked.

hope your system follow the specification.

glibc-2.4 returns EBUSY if mutex locked, let use this adventage.

however i have no idea if this behavior will persist in later glibc releases. seems to be not very clever to use it ...
